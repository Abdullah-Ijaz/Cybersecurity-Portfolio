Pickle Rick

TryHackMe difficulty: easy unguided web · linux

$ ./objective

Objective

A single Linux web host running Rick Sanchez's improvised web app. The goal was to compromise the machine and recover the three "secret ingredients" scattered across the filesystem — each one gated behind a different layer: web enumeration, a command-execution panel with weak input filtering, and finally a sudo misconfiguration granting full root. No walkthrough; findings below are the path I took start to finish.

$ ls ./tools

Tools Used

Nmap Gobuster curl Browser / View-Source base64

$ ./methodology --verbose

Methodology

01 · Reconnaissance

I started with a service/version scan and saved the output for reference. Only two ports were open — a small attack surface that pointed straight at the web app.

terminalnmap
$ nmap -sV -sC -oN initial_scan.txt <TARGET_IP>
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH (Ubuntu)
80/tcp open  http    Apache httpd

SSH (22) is a likely pivot point but useless without credentials; Apache on 80 was the way in.

02 · Web Enumeration

I brute-forced directories against the web root with Gobuster. Most paths returned 403 Forbidden, but two were reachable and interesting: login.php and robots.txt.

terminalgobuster
$ gobuster dir -u http://<TARGET_IP> -w /usr/share/wordlists/dirb/common.txt
/index.html      (Status: 200)
/login.php       (Status: 200)
/robots.txt      (Status: 200)
/assets          (Status: 301)
# most other paths: 403 Forbidden

Two findings gave me a credential pair. The username was sitting in an HTML comment in the homepage source (a "note to self" left in the markup), and robots.txt held a single odd string that turned out to be the password.

terminalcurl
# password — from robots.txt
$ curl http://<TARGET_IP>/robots.txt
Wubbalubbadubdub

# username — from the homepage HTML source comment (View Source / Ctrl+U)
<!-- Note to self, remember username! R1ckRul3s -->

03 · Exploitation — the command panel

Logging into login.php with R1ckRul3s : Wubbalubbadubdub dropped me onto a "Command Panel" — effectively a web shell that runs my input as OS commands on the server. whoami and ls -a confirmed code execution as the web user and revealed a suspicious file.

The panel blocks cat — but blocklist filtering like that is trivial to step around, since dozens of other utilities read files. I used less to pull the contents of the first ingredient file:

command panelfilter bypass
less Sup3rS3cretPickl3Ingred.txt
# cat is denied; less / more / head / tail / grep . all read the file just fine

→ First ingredient recovered: redacted — solve it yourself :)

04 · Privilege Escalation — sudo NOPASSWD

A Clue.txt nudged me to keep hunting the filesystem. The decisive check was sudo rights — and the web user could run anything as root without a password:

command panelsudo -l
sudo -l
User www-data may run the following commands:
    (ALL) NOPASSWD: ALL

That's game over for the host — (ALL) NOPASSWD: ALL means I can read or do anything as root. The second ingredient lived in /home/rick/ in a file with a space in its name. The browser mangled the raw output, so I read it through base64 — a clean trick for forcing awkward bytes through a web display — then decoded it locally:

command panelbase64 read
sudo base64 "/home/rick/second ingredients"
MSB...redacted...Cg==
# decode locally:  echo 'BASE64' | base64 -d

→ Second ingredient recovered: redacted

The third is almost always in root's home — so I looked there with my sudo rights and read the final file:

command panelroot access
sudo ls -la /root
-rw-r--r-- 1 root root  ...  3rd.txt
sudo less /root/3rd.txt

→ Third ingredient recovered: redacted. All three ingredients collected — box complete.

$ ./mitigation

Mitigation

Every step here maps to a fixable defensive failure. In priority order:

  • ▹ Never grant a web/service account passwordless full sudo. www-data with (ALL) NOPASSWD: ALL turns any code execution into instant root. Scope sudo to specific, non-shelling binaries and require a password — least privilege is the single most important fix here.
  • ▹ Don't build a command-execution panel — and don't rely on a blocklist if you do. Filtering cat while allowing less, grep, base64, etc. is security theater. Command execution from user input should be removed entirely, or replaced with a fixed allowlist of parameterized actions.
  • ▹ Keep secrets out of client-visible surfaces. Credentials in HTML comments and in robots.txt are readable by anyone. robots.txt is a crawler hint, never an access control.
  • ▹ Enforce real authentication and rate limiting on the login, and store no plaintext reusable passwords. The single hard-coded credential pair gated the entire app.

$ ./takeaway

Takeaway

A tidy demonstration of chaining low-severity findings into total compromise: information disclosure (source + robots.txt) → authenticated command execution → sudo misconfiguration → root. No single bug was exotic, but stacked together they were fatal — which is exactly how real internal footholds tend to look.
